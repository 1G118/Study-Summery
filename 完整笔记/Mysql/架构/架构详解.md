# 1.连接层
连接层在MySQL占比中并不重要，他只负责连接和分配权限，只需记住MySQL是基于TCP协议进行传输即可
>存在可连接的最大客户端数量。默认是151，可以更改。

大致流程如下

```
1. 客户端发起TCP连接请求
       ↓
2. 服务器接受连接（三次握手）
       ↓
3. 服务器发送握手包（包含认证方式）
       ↓
4. 客户端发送用户名、密码
       ↓
5. 服务器验证身份
       ↓
6. 验证通过，分配线程处理后续请求
```

# 2.服务层
这层涉及到解析，执行SQL语句[[操作]]。
> 缓存在8.0版本后就被移除

## 1.解析器
在你输入SQL语句后，MySQL并不会立即执行。他会首先将SQL语句进行解析。
### 词法分析
将你的输入划分为关键词和非关键词，关键词就是语法词，如select，from，to等。非关键字例如你输入的表名和列名等。还有一部分是运算符，常量等。
例如：
```
SELECT * FROM users WHERE id = 1
关键字: SELECT
运算符: *
关键字: FROM
表名: users
关键字: WHERE
列名: id
运算符: =
常量: 1
```

### 语法分析
根据词法分析的结果会判断你输入的MySQL语法是否正确。
-  语法错误：SLECT -> SELECT


## 语法树
语法分析后，MySQL会构建一颗**语法树（Abstract Syntax Tree, AST**这棵树是SQL语句的结构化表示，后续的优化器、执行器都依赖这棵树来工作。

| 作用    | 说明                        |
| ----- | ------------------------- |
| 结构化表示 | 将SQL文本转化为程序可以理解的数据结构      |
| 提取信息  | 方便获取SQL类型，表明，字段名，WHERE条件等 |
| 语义检查  | 预处理器基于语法树进行语义分析           |
| 优化基础  | 优化器基于语法树进行查询优化            |
语义检查和优化基础都是语法树方便后续模块使用，避免一个动作重复进行造成性能的浪费，提高效率。如果你学过Java，你可以将这步理解为转成class文件。

## 2.预处理器
>语义分析和验证

当你的语句走到这步的时候，恭喜你，证明你已经基本掌握了MySQL的语法，不然在解析器部分就会被打回。后续就要检查你的语义是否正确。
主要是
- 检查你的SQL语句里表或字段名是否存在
- 将 * 扩展为表上的所有列
- 验证数据类型是否匹配
在查询的同时还会判断你是否有访问权限，若权限不够就会返回error
```
SELECT * FROM orders;
ERROR 1142 (42000): SELECT command denied to user 'test'@'localhost' for table 'orders'
```

## 3.优化器
>生成最优的执行计划

这步我认为是比较重要的，这部分就可以解释为什么MySQL会判断何时使用二级索引而不是主键，为什么我们要创建复合索引。
优化器主要进行了这几步
- 索引选择
- 连接顺序优化
- 等价替换
- 覆盖索引
- 执行计划
优化器是如何来判断用哪种方式运行更快的？
1. 给不同的[[操作]]给予不同的权重

| 常数                | 默认值  | 说明        |
| ----------------- | ---- | --------- |
| disk_read_cost    | 1.0  | 读一个磁盘页的成本 |
| memory_read_cost  | 0.25 | 读一个内存页的成本 |
| row_evaluate_cost | 0.1  | 评估一行数据的成本 |
| key_compare_cost  | 0.1  | 索引键比较的成本  |
2.  统计依赖表信息
3.  选择率=满足条件的行数 / 总行数
4.  基数 = 列中不同值的数量
5. 数据分布
>要稍微注意的是选择率其实越低越有价值。因为选择率越少说明满足条件的行数很少，那么只需要扫描几行就可以找到想要的结果

## 4.执行器
>执行优化后的查询计划

这步没有什么值得注意的了解大致流程即可：
```
执行计划
    ↓
初始化执行环境
    ↓
调用存储引擎接口
    ↓
获取数据
    ↓
处理数据（JOIN、排序、分组）
    ↓
返回结果集
```

## 5.日志系统
### 慢查询

```
-- 开启慢查询
SET GLOBAL slow_query_log =ON;

-- 设置慢查询阈值（秒）

SET GLOBAL long_query_time = 2;

-- 查看慢查询日志位置

SHOW VARIABLES LIKE 'slow_query_log_file';
```
**慢查询（Slow Query）** 是指执行时间超过指定阈值的SQL语句。用于查询性能瓶颈和SQL问题。可以理解为一个检测器，开启后所有运行时间超过你设定的SQL语句都会被记录在慢查询日志里。
日志格式大致是：
```
# Time: 2024-01-15T10:30:45.123456Z
# User@Host: root[root] @ localhost []  Id: 123
# Query_time: 15.234567  Lock_time: 0.000123  Rows_sent: 1  Rows_examined: 100000
SET timestamp=1705312245;
SELECT * FROM orders WHERE user_id = 123 AND status = 'pending';
```
字段说明：

| 字段            | 说明         |
| ------------- | ---------- |
| Time          | 查询执行的时间    |
| User@Host     | 执行查询的用户和主机 |
| Query_time    | 查询总耗时（s）   |
| Lock_time     | 等待锁的时间     |
| Rows_examined | 扫描的行数      |
| Rows_sent     | 返回的行数      |
| 最后一行          | 执行的SQL语句   |
SET timestamp=1705312245是**Unix时间戳**，表示查询执行的精确时间。

慢日志是txt文件，阅读起来会很麻烦，需要逐行逐行的看，可以去使用MySQL提供的mysqldumpslow工具
用法举例：
```
# 按查询时间排序，显示前10条
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 按扫描行数排序
mysqldumpslow -s r -t 10 /var/log/mysql/slow.log

# 按次数排序
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log
```

| 参数  | 说明                         |
| --- | -------------------------- |
| -s  | 排序方式（t=时间，c=次数，r=行数，l=锁时间） |
| -t  | 显示前N条                      |
| -g  | 过滤包含指定模式的查询                |
不过他依旧不是很主流的选择
比较常用的是**pt-query-digeset**
它是 Percona Toolkit 中一个用于分析 MySQL 查询的性能分析工具，主要用来定位慢查询、优化数据库性能。它对比mysqldumpslow的优点是它可以直接生成一份完整的分析报告。这里如何使用不做过多讲解。

但无论是mysqldumpslow还是pt-query-digest，它们都需要你主动去查询是否出现问题，换句话说当你去使用它们的时候就证明MySQL已经出现问题了。你也不想你的数据库在软件运行过程中，中途暴毙吧？那要怎么办呢？——长效监控
1. 接入监控工具，如Prometheus + Grafana
2. 在合并代码的时，把慢查询也加入评判的过程，让提交者要写明相关SQL代码的explain分析。

>除开慢查询日志外，还有通用日志和错误日志，详细会在日志篇进行说明，在此就不多做赘述。


# 3.存储引擎层
>存储引擎是MySQL中负责数据存储和提取的底层组件，它决定了数据如何存储、如何建立索引、如何锁定数据等。

这部分就很重要了，难点也是这个部分，像索引的实现，锁，事务，并发等这些功能都出自这部分，这部分不会对这些点细讲，更多的是介绍InnoDB的架构。（这里需要提一嘴的是数据并不是存储在这部分，存储引擎重点在引擎二字，可以将他理解为管家，当你提出要找某件物品时，由于数据是由它整理的，所以它能更快的找到，数据还是存储在你本地硬盘里。）
