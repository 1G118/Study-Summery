Buffer Pool（缓冲池）用于提升查询的性能。

>在学习Buffer Pool之前，我们需要了解为什么数据库要在内存部分设计这么一个部分。

	在计算机当中，对硬盘里的内容进行读写（IO）是很慢的，速度可能是在毫秒级别。虽然在日常生活中可能觉得毫秒级别已经足够快了，但内存进行读写的速度是可以到达纳秒的，它们之间的差距大概是在几万倍。这样一对比就明显能感受到计算机在进行硬盘操作时是有多慢了。
	性能高的同时往往就代表会在其他地方进行取舍，内存是无法做长期存储的，每次我们开关机电脑，内存里面记录的内容都会消失（一些提到持续性，往往指的是在开机时，软件或程序会自动将之前的内容从硬盘里读取出来）。且一台电脑的内存大小比起硬盘往往要小很多，内存基本都是8g，16g，32g。而硬盘基本就是512g，1t，2t。所以数据库里的内容，靠内存的大小是无法全部存储的，并且数据库的内容都是需要长期保存的。数据库的内容基本还是存储在硬盘里。
	这里就会疑惑，既然用硬盘会导致读取变慢，内存又太小且无法做到长期保存，那是否会什么办法可以中和它两的优势？这就是Buffer Pool设计的初衷。
	为了提高速率，设计者决定在内存部分单独开辟一个部分作为缓冲池，也叫热数据存储区。它横插进了读写命令和硬盘两者之间，当一个读命令进来时，他会优先经过缓存池，查询里面是否已经有了记录如果有则直接将这个结果返回。如果没有，那么再到硬盘里进行查询，找到后将这个数据入缓冲池，最后返回结果。这样如果一个经常被访问的数据只有第一次访问的时候需要进行IO，之后同样的操作在内存部分就会直接返回，极大的提升了速度。这个缓冲池也就是我们常说的Buffer Pool。

同样还有很多其他的问题，比如：
	1.  既然内存大小有限，如果有大量查询不同数据的查询指令，将Buffer Pool的内存占满，此时又来了一个查询会发生什么？
	2.  数据是以怎样的方式写入Buffer Pool。
	3.  数据修改后，在Buffer Pool和硬盘当中是如何实现一致性的？
在了解Buffer Pool是如何解决这些问题之前是需要先学会[[InnoDB的磁盘结构]]

在了解完磁盘结构之后就会了解到Buffer Pool除开内存部分外还有一部分在硬盘做长期存储的作用（两者其实不是一个东西，但记录的内容差不多，所以这里将两者归为一类了），平时经常被提到的Buffer Pool指的是内存上的，但在硬盘部分叫Change Buffer。

提到Buffer Pool逃不开的就是LRU算法，在InnoDB里LRU算法是不足以满足管理需求的，因此被优化成了LRU List链表结构来管理缓冲页。在了解这部分之前要先知晓Buffer Pool的内部结构
>但除LRU List之外，Buffer Pool还通过Free List，Flush List来管理链表。

# 2. 内部结构
## 2.1 缓存页
还记得磁盘结构里表是怎么被划分的吗？表->段->区->页->行。
Buffer Pool被划分成的是一个个与磁盘页大小相同的页框（16kb），每个页框可以缓存一个磁盘数据页。
>为什么是页而不是区或者段？如果是认为区或段太大了，那为什么不是行呢？偏偏是页？还记得磁盘机构部分提到的，InnoDB读写的最小单位是什么？——页。    

## 2.2 控制块
每个缓存页前面有一个控制块，记录该页的元信息。缓存页区域和控制块区域是分开的，但是在同一连续内存中。
元信息：
	1. 页所属表空间的ID和页号
	2. 页的类型
	3. LSN
	4. 是否为脏页
	5. 锁定信息
	6. Free List/LRU List/Flush List 的链表指针

## 2.3 三大链表
Buffer Pool通过三个链表来管理所有的缓存页，不是一个单独的数据结构，这三个链表都存储在控制块里，这里单独拧出来是因为比较重要。（都是双向链）

| 链表         | 作用                                     |
| ---------- | -------------------------------------- |
| Free List  | 存放空间的缓存页控制块。刚启动时所有页都在，需要加载磁盘页时，从这里取空闲页 |
| LRU List   | 存放已使用的缓存页，，淘汰冷数据，保留热数据                 |
| Flush List | 存放脏页（内存中已修改但未刷盘的页），后台线程按顺序将脏页刷回磁盘      |

想要从代码结合设计结构了解链表是怎么工作比较苦难的话在理解上就将链表单独作为一个部分去理解会好很多，影响也不大。
